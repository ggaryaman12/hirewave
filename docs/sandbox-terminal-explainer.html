Terminal layer explainer

What we built around the candidate terminal

The candidate terminal is not your Mac terminal. It is a controlled command runner that decides what can run, what files it runs against, what evidence gets saved, and how honestly the report labels the result.

In plain language

This is the mental model to keep in mind before reading code.

1

The terminal is a gate

When a candidate types npm test, the app does not blindly run any shell command. It sends the request through a controlled server route and a provider.

2

The files are fingerprinted

Before running the command, the app creates a workspace manifest: file paths, sizes, hashes, and one digest for the whole mounted workspace.

3

The report stays honest

The report says whether the command came from a demo provider, local-dev runner, or future production provider. It should not pretend demo evidence is production isolation.

Three provider modes

Provider means "who actually handles this command". Today only the simulated path is default for demos.

Default

Simulated provider

This is the safe MVP demo path. It does not run arbitrary host shell commands. It supports only the controlled commands used by the assessment room.

Readinessdemo_only
Isolationnone
Real code executionnot_supported

What gets stored for reviewers

The important change is not only "tests can run". The important change is "we can prove what kind of execution produced the result".

CommandRun The compact DB row: command, status, output, exit code, start time, finish time.
TestResult Parsed test names and pass/fail status, especially for npm test.
SessionEvent Append-only telemetry for command started, output, finished, test finished, and sandbox snapshots.
FileSnapshot Candidate edits and safe runner-owned snapshots that should appear in final diffs.
Report metadata Provider kind, readiness, capability gaps, workspace digest, output truncation, cleanup, and risk flags.

            Done now
            
Foundation and evidence layer

            
    
              
  • Provider interface exists.
    •

            Not done yet
            
Real production sandbox

            
    
              
  • No real E2B, Daytona, CodeSandbox, Docker, or microVM adapter yet.
    • 
              
  • No production per-session credential boundary yet.
    • 
              
  • No enforced production network policy yet.
    • 
              
  • No background lifecycle cleanup/reaper yet.
    • 
              
  • No enterprise audit UI for raw sandbox runs yet.
