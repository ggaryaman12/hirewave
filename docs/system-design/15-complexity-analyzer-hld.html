Complexity Analyzer & Simulator — High-Level Design

"Analyze Complexity" button → measures Big-O of the student's code, explains it line by line. ← hub

00  The plain idea (read this first)

A student writes a solution in the editor. They click one button. We tell them: how slow it is (time Big-O), how much memory it uses (space Big-O), and why — by walking through the code like a debugger and pointing at the expensive lines.

Think of it like a fitness tracker for code. The student runs their solution; we strap a step-counter on it, run it on inputs that grow bigger and bigger, and watch how fast the work grows. If doubling the input roughly doubles the work → that's O(n). If doubling it quadruples the work → O(n²). We literally measure it, then explain it.

01  The one hard truth (your best interview line)

You cannot prove the Big-O of arbitrary code just by reading it. That is mathematically impossible — it reduces to the halting problem (Rice's theorem). Any tool claiming to "statically prove" complexity for all code is lying.

So the whole design flips: measurement is the truth, reading-the-code is only the explanation.

• Dynamic profiling (running the code on growing inputs) = ground truth for the Big-O number.
• Static analysis (parsing the code into a tree) = the storyteller that says "this line runs ~n² times because it's a loop inside a loop." It labels and explains; it does not decide the final answer.

03  The request flow (two-phase — the key UX move)

Static is instant; dynamic is slow (it runs code many times). So we answer twice: fast then accurate.

1. Clicksend {problemId, lang, code}
2. Static, <300msparse → annotate lines → paint NOW
3. Enqueuedynamic job by code-hash
4. Measurerun n=8,16,32… curve-fit
5. Stream backreal Big-O + simulation steps
The editor lights up with per-line notes immediately from the static pass. The measured Big-O and the step-through debugger arrive a moment later and replace the guesses. Student never stares at a spinner.

04  What comes back (the result shape)

interface ComplexitySummary {
  bigOTime:    string;   // "O(n^2)"  — MEASURED, the real answer
  bigOSpace:   string;   // "O(n)"
  confidence:  number;   // curve-fit R² (0–1) — honesty meter
  staticGuess: { time: string; space: string };  // from reading the code
  explanation: string;   // "outer loop n × inner loop n → n²"
  hotspots:    { line: number; share: number }[];   // expensive lines
  perLine:     LineAnnotation[];   // one note per editor line
  suggestion?: string;   // "use a hash map → O(n)"
}
interface LineAnnotation {
  line: number; executed: string;   // "~n times"
  timeFactor: string; spaceImpact: string;
  note: string; inLoop: boolean; inRecursion: boolean;
}
confidence is the trust feature. We show "measured ~O(n²) (high confidence)", never a bare claim. If we ran wrong code, showing a confident wrong answer is worse than no feature — so we always expose how sure we are.

07  🎤 Interview scripts

"Design a LeetCode-style complexity visualizer."
Start with the hard truth: you can't statically prove Big-O for arbitrary code, so the system is measurement-first. Instrument the code, run it across growing input sizes in a sandbox, count operations, and curve-fit the growth to get the empirical Big-O with a confidence score. Use a parsed AST only to attach per-line explanations and detect loop nesting and recursion. Two-phase response: instant static annotations, then the measured result streams in.
"How do you estimate complexity of arbitrary user code?"
I don't trust reading the code — that's undecidable. I measure: run on n = 8, 16, 32… count ops per run, fit the points against a ladder of candidate curves (1, log n, n, n log n, n², 2ⁿ), and pick the best fit by R². The AST is only the storyteller that explains the shape.
"How do you stop Analyze from killing the judge under load?"
On-click only with debounce, never per-keystroke. Cache results by code-hash so repeats are free. Give the analyzer its own worker pool and queue, isolated from Run/Submit, so a contest spike of Analyze clicks degrades into queue depth, not failed submissions.

08  Check yourself

Q1. Why is dynamic the source of truth, not static?
Because inferring exact Big-O from arbitrary source is undecidable (halting problem). Running the code and measuring growth is concrete; reading it can only guess and explain.
Q2. Why answer in two phases?
Static is instant but approximate; dynamic is accurate but slow. Paint static annotations immediately so the UI is responsive, then replace them with the measured result when the worker finishes.
Q3. Why a separate worker pool from the judge?
So a flood of Analyze clicks can't starve real Run/Submit traffic. Isolated lanes = a spike in one becomes latency there, not failures everywhere.
