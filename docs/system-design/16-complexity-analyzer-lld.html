Complexity Analyzer — Low-Level Design

The three engines: static reader, dynamic measurer, line-by-line simulator. ← hub · ← HLD

00  The plain idea (read this first)

Three modules, each with one job:

A · Static engine — reads the code into a tree, spots loops and recursion, writes "this line runs ~n times" notes. Fast, runs instantly, only explains.

B · Dynamic profiler — actually runs the code on bigger and bigger inputs, counts the work, and fits a curve to find the real Big-O. This is the truth.

C · Simulator — runs the code once on a tiny input and records every step (which line, what the variables are, how deep the recursion is) so the student can step forward/back like a debugger.

01  A · Static engine — the storyteller

Lives in lib/complexity/static/. Pure functions, no running of code. Powers the instant first paint.

type Lang = 'cpp' | 'java' | 'js';

parseCode(src, lang): NormalizedAst       // tree-sitter → uniform tree
buildControlFlow(ast): ControlFlowGraph   // blocks + jumps
buildCallGraph(ast):   CallGraph          // who calls whom (recursion)
inferComplexity(cfg, cg): ComplexitySummary
annotateLines(summary, ast): LineAnnotation[]
Parser = tree-sitter (one grammar each for C++, Java, JS). Why it, not a real compiler? It's incremental, runs in Node via WASM, has a uniform node API across languages, and — crucially — it's error-tolerant, so it still parses half-written code while the student types.

02  A · How the Big-O guess is computed (the algebra)

Represent every chunk's cost as a symbolic term like c·n^a·(log n)^b, then combine bottom-up:

Code shapeRuleResult
Statements in a rowtake the max (biggest wins)O(n)+O(1) → O(n)
Loop inside a loopmultiply the bounds (nesting depth)n × n → O(n²)
if / elsemax of the branches (worst case)worst arm
Halving loop i*=2that loop is log nO(log n)
Recursion T(n)=a·T(n/b)+f(n)Master theorem / Akra–Bazzie.g. mergesort → O(n log n)
Tree recursion (2 calls, n-1)exponential flagnaive fib → O(2ⁿ)
Library call (sort, map.find)lookup table of known costssort → O(n log n)
The collection-ops table is keyed by (language, type, method): unordered_map.find→O(1), map.find→O(log n), Arrays.sort/std::sort/Array.sort→O(n log n). This same table powers the "suggest a hash map" hint when it sees nested loops doing lookups (the classic 2-sum fix).

03  A · What static gets WRONG (be honest)

Always say the limits out loud — it's the mark of someone who understands the problem.

  • Data-dependent loops (while (x != target)) — bound unknown.
  • Early exits / amortized costs — looks O(n²), really O(n).
  • Complexity hidden inside library calls we didn't model.
  • Indirect / virtual recursion through interfaces.
Fallback rule: any unknown bound on a hot path → static reports "≥ O(n) (undetermined)" with a lowConfidence flag. Static never claims a tight bound it can't justify. That's exactly the gap the dynamic measurer fills.

04  B · Dynamic profiler — the truth machine

Lives in lib/complexity/dynamic/, runs inside the analyzer worker + sandbox.

Step 1 — Instrument the code (add a step-counter):
• JS: rewrite the code (babel) to do __ops++ on every statement and __hit[line]++ per line. Precise.
• C++ / Java: insert a counter at the top of each basic block, compile + run with the existing harness. Coarser (block-level) but enough for the curve shape and hotspots. Peak memory comes from the sandbox's existing MLE metering. 
// Step 2 — run on a growing ladder of input sizes
ladder = [8, 16, 32, 64, 128, 256 … up to cap]
for (n of ladder) {
  input = genInputForSize(problem, n)          // reuse in-repo generators
  { ops, peakMem } = runInstrumented(code, input, limits)
  record(n, ops, peakMem)
}
// hit a time/op/mem cap → stop early, fit on what we have, flag truncated

05  B · Curve-fitting — turning points into Big-O

We have points (n, ops). Which curve do they follow?

Try each candidate — 1, log n, n, n log n, n², n³, 2ⁿ — fit ops ≈ c·model(n) by least squares, keep the best R². That R² becomes the confidence shown to the student. Same fit on peakMem(n) gives space Big-O.

The famous trap: n vs n log n look almost identical on a small ladder. Don't trust R² alone — use the ratio test: compute ops(n)/candidate(n) across the ladder; for the true model that ratio stays flat. Pick the model with the lowest ratio-variance. 
reconcile(static, dynamic): ComplexitySummary
  agree           → high confidence, cite both
  disagree        → trust DYNAMIC, explain the gap:
                    "structure looks O(n²) but measured ~O(n log n)
                     — likely an early exit or a library sort"
  static unknown  → dynamic stands alone
  dynamic skipped → static only, low confidence

06  C · The simulator — debugger feel

interface Step {
  idx: number; lineNumber: number;     // 1-based, aligned to editor
  variablesSnapshot: Record<string,string>;
  callStack: { fn: string; line: number }[];
  note?: string;                       // "inner loop body, ~n² times total"
  iterationIndex?: number; recursionDepth: number;
}
How steps are made: run ONE small input (n=8), instrumented to emit events (enter-line, write-var, push/pop-call). The worker saves these as SimulationStep rows. Each step shows both "where am I now" (from the run) and "how often does this line run as f(n)" (from the static notes).

Editor integration (Monaco): step.lineNumber maps to the real editor line (the instrumenter never reformats, so line numbers stay aligned). Client uses deltaDecorations to highlight the current line, view zones for the floating bubble, and a side panel for call stack + variables. Stepping forward/back is pure client state over the downloaded steps — zero server calls per step.

07  C · Runaway code — infinite loops & deep recursion

Same caps as the profiler, plus a MAX_STEPS ceiling (e.g. 5000) and a recursion-depth limit. Exceed any cap → stop, return the partial trace + { truncated:true, reason:'step_cap' }. UI: "simulation truncated at step 5000 — likely unbounded; static estimate still shown." The student still learns something instead of getting an error.

08  Data model & API contracts

model ComplexityAnalysis {
  id String @id @default(cuid())
  codeHash String @unique     // sha256(code+lang+problemId) — the cache key
  problemId String;  lang String
  status String               // queued|static_ready|done|failed|truncated
  bigOTime String?;  bigOSpace String?;  confidence Float?
  summary Json?               // ComplexitySummary
  steps SimulationStep[]
  @@index([problemId])
}
model SimulationStep {
  id String @id @default(cuid())
  analysisId String;  idx Int
  lineNumber Int;  vars Json;  callStack Json
  note String?;  recursDepth Int
  @@unique([analysisId, idx])
}
POST /api/dsa/complexity   {problemId, lang, code}
  → {analysisId, status:'static_ready', summary}   // instant paint
GET  /api/dsa/complexity/:id
  → {status, summary}                              // poll/SSE until 'done'
GET  /api/dsa/complexity/:id/simulation?cursor=0&limit=200
  → {steps[], nextCursor, truncated}

09  🎤 Interview scripts

"Static says O(n²), measurement says O(n log n). What do you show?"
Trust the measurement for the label — dynamic is ground truth. The disagreement isn't a bug, it's a teaching signal: surface it as "structure suggests O(n²) but it measured O(n log n), likely an early exit or a library sort." Reading explains; running decides.
"How do you tell O(n) from O(n log n) empirically?"
R² alone fails because the curves look alike on small inputs. Use a ratio test: divide measured ops by each candidate model across the input ladder; the true model gives a flat ratio. Pick the lowest ratio-variance, not just the highest R².
"How do you instrument C++/Java safely for per-line counts?"
Source-transform a counter into each basic block, then compile and run inside the existing sandbox with hard op, time, and memory caps. It's block-level, coarser than JS statement-level, but enough for curve shape and hotspots; peak memory reuses the judge's MLE metering.

10  Check yourself

Q1. Why tree-sitter over a real compiler front-end?
Uniform node API across languages, runs in Node via WASM, incremental, and error-tolerant so it parses half-written code while the student types.
Q2. What does the Master theorem do here?
Solves recursion cost: from T(n)=a·T(n/b)+f(n) it gives the closed-form Big-O — e.g. mergesort's 2T(n/2)+O(n) → O(n log n). Used in the static pass to label recursive functions.
Q3. Why capture the simulation from a tiny input, not the big ones?
A full trace explodes in size and is unreadable. A small n (≈8) gives a short, followable step sequence; the static notes already tell the student how often each line runs as a function of n.
