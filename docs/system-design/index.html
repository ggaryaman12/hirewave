🎤 Interview cheat-sheet (2-sentence pitches)

Why a queue?
Decouples arrival rate from service rate. A 100× contest spike becomes queue depth (graceful latency), not blocked threads and 503s — and the same buffer turns a Judge0 outage into delayed results instead of failures.
Async worker pattern
Web tier just authenticates, validates, enqueues; a separately-scalable worker pool does the slow work. Scale throughput by adding workers, independent of the API tier.
Idempotency
Every submission carries a unique idempotencyKey; retries and double-clicks collapse onto one row. One action = exactly one Judge0 run.
Thin payload / claim-check
Queue message carries only IDs; the worker loads truth from Postgres. Small messages, no stale data.
At-least-once + idempotent consumer
Crash after Judge0 but before ack → redelivery. The worker refuses to overwrite a DONE submission, so reprocessing is safe.
Error classification
Property of the submission (compile error, WA, TLE) → valid verdict, no retry. Property of the infra (5xx, timeout) → retry with backoff, then DLQ.
Two Judge0 timeouts
API-response timeout = is the service alive (infra → retry); wall-clock cap = is the user's code too slow (→ TLE verdict). Don't conflate them.
Result delivery
202 + jobId, browser polls with backoff to DONE/ERROR; upgrade to SSE so the server pushes the verdict and kills the wasted polls.
AI orchestration
One orchestrator behind a provider abstraction owns routing, timeouts, retries, caching, circuit breaker, and cost metering; adding an AI feature is a route config, not new glue.
Multi-tenancy
Row-level tenantId injected by middleware + Postgres RLS as the database-enforced backstop, so a forgotten filter can't leak. Escalate to DB-per-tenant only for enterprise.
Profile stats
Cheap read-your-write metrics update on write via an outbox (no drift); global/expensive ones (percentile) precompute offline + cache; leaderboards use a Redis sorted set.
Observability
Correlation ID threaded through every hop turns an async flow into one queryable timeline; RED on the API, USE on workers and Judge0; alert on queue depth, Judge0 p95/errors, DLQ growth.
Security
Sandbox untrusted code with hard CPU/mem/time limits and no network; auth + per-user/IP/tenant Redis rate limits protect the exec and AI APIs from abuse.
Reliability trio
Timeouts free threads, retries+backoff self-heal, idempotency makes retries safe — only correct together. Circuit breakers stop hammering a dead dependency.
Complexity analyzer
You can't statically prove Big-O for arbitrary code (undecidable), so measure it: instrument the code, run on growing input sizes, curve-fit the operation counts to get empirical Big-O with a confidence score. The parsed AST only explains and annotates lines; running the code decides the answer.

🔁 Reusable patterns (apply these to ANY system-design question)

  • Async worker — slow/spiky work → enqueue + worker pool (judge, video encode, email, ML inference)
  • Claim-check — big payload → store + pass a reference
  • Idempotency key — safe retries for any side-effecting op
  • Outbox — atomic "do X + emit event" without distributed transactions
  • CQRS / read model — precompute expensive reads, serve O(1)
  • Circuit breaker — stop hammering a failing dependency
  • Token bucket — burst-tolerant rate limiting, shared in Redis
  • Belt + suspenders — convenience layer (app filter) + enforcement layer (RLS); recurs everywhere
  • RED + USE — what to measure: requests vs resources
